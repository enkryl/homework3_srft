# Arithmetic Puzzle: сравнение GRPO, SFT→GRPO и SRFT

В этом проекте сравниваются три подхода к обучению модели на арифметической задаче:

- **GRPO-only**
- **SFT → GRPO**
- **SRFT** (single-stage схема по мотивам статьи)

Задача: по набору чисел построить корректное арифметическое выражение, используя каждое число ровно один раз, чтобы получить заданный target.

## Структура проекта

- `check_hard_v2.ipynb` — проверка среды и генерация/валидация данных
- `GRPO_full_experiment.py` / `GRPO_full_experiment.ipynb` — основной запуск всех экспериментов
- `srft_paper_module.py` — вспомогательная реализация SRFT-подобного обучения
- `envs/target24/` — реализация среды и verifier
- `data_v2/` — train / val / test сплиты и данные по уровням
- `target24_experiment_report.ipynb` — ноутбук с итоговым отчётом и графиками

## Как воспроизвести результаты

### 1. Проверка среды и данных

Открой `check_hard_v2.ipynb`.

Этот ноутбук:
- проверяет корректность среды и verifier;
- формирует и валидирует датасеты в `data_v2/`;
- создаёт train / val / test сплиты, а также разбиения по уровням сложности `L1`–`L10`.

Основные файлы:
- `data_v2/train.jsonl`
- `data_v2/val.jsonl`
- `data_v2/test.jsonl`
- `data_v2/train_L*.jsonl`, `val_L*.jsonl`, `test_L*.jsonl`
- `data_v2/sft_gold.jsonl`

### 2. Обучение моделей

Основной эксперимент запускается из `GRPO_full_experiment.py`
(или из одноимённого ноутбука, если используется notebook-версия).

В файле реализованы три режима обучения:
- **GRPO-only**
- **SFT → GRPO**
- **SRFT**

Скрипт:
- загружает базовую модель;
- запускает обучение по curriculum;
- оценивает модель на уровнях `L1`–`L10`;
- сохраняет метрики `pass@k`.

Артефакты обучения сохраняются в:
- `logs_*` — логи, json-файлы с метриками и графики;
- `runs_*` — LoRA-адаптеры и промежуточные чекпоинты.

Примеры:
- `logs_grpo_only/`
- `logs_sft_grpo/`
- `logs_srft/`
- `runs_grpo_only/`
- `runs_sft_grpo/`
- `runs_srft/`

### 3. Анализ результатов

После обучения можно запустить блок анализа в `GRPO_full_experiment.py`
или открыть готовый отчёт:

- `target24_experiment_report.ipynb`

В отчёте собраны:
- графики `pass@1` и `pass@128`;
- сравнение методов по уровням сложности;
- примеры решённых и нерешённых задач;
- интерпретация результатов;
- выводы по поведению **GRPO**, **SFT→GRPO** и **SRFT**.

## Основной вывод

По результатам эксперимента лучший метод — **SFT → GRPO**:
он показывает наиболее устойчивое качество и лучше переносится на сложные уровни.

**GRPO-only** обучается слабее и заметно проседает на трудных примерах.

**SRFT** в текущей реализации даёт результаты хуже ожидаемых.
По логам видно, что в self-rollout части почти отсутствуют положительные on-policy примеры, поэтому метод не получает достаточно сильного сигнала для обучения. Из-за этого SRFT в данном эксперименте уступает схеме **SFT → GRPO**.

## Примечание

Папка `envs/target24/` сохраняет старое техническое имя среды, однако сама задача в проекте — это не классический Target24, а более общая **арифметическая задача с заданным target**.
